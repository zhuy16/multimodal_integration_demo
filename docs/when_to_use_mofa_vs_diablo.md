# When to Use MOFA+ vs DIABLO

A practical decision guide for multi-omics integration in translational research.

---

## The Core Question

Both MOFA+ and DIABLO integrate multiple omics layers from the same set of samples, but they address fundamentally different scientific questions:

- **MOFA+** asks: *"What are the major axes of biological variation in my multi-omics data, and which platforms contribute to them?"*
- **DIABLO** asks: *"What is the smallest multi-omics signature that reliably predicts my clinical outcome?"*

The choice flows directly from whether you have a response variable and what you want to do with results.

---

## Decision Flowchart

```
Do you have a well-defined outcome/label for all samples?
│
├── NO ──► Do you have missing data (not all samples profiled on all platforms)?
│          ├── YES ──► MOFA+ is the best choice
│          └── NO  ──► MOFA+ or multi-omics factor analysis; consider
│                       UMAP/tSNE on concatenated data as baseline
│
└── YES ──► How many samples per class?
            ├── < 20  ──► Neither method is reliable; consider regularized
            │              single-omics or literature-guided pathway scoring
            ├── 20–50 ──► DIABLO with strong regularization (small keepX);
            │              validate carefully with leave-one-out CV
            └── > 50  ──► DIABLO is well-powered; tune keepX via perf()
```

## Detailed Comparison

### MOFA+: Multi-Omics Factor Analysis

**Mathematical foundation:**
MOFA+ is a Bayesian latent factor model. It assumes that the observed data matrix **X** (samples × features, per view) is generated by a small number of latent factors **Z** (samples × K) weighted by view-specific loadings **W** (features × K):

```
X_m = Z · W_m^T + ε_m
```

Each factor captures a linear combination of features across views that explains a portion of the total variance. The Bayesian treatment with Automatic Relevance Determination (ARD) priors drives irrelevant factors to zero, effectively selecting the number of active factors automatically.

**Key strengths:**
- **Missing data tolerance**: MOFA+ marginalizes over missing entries during inference. If 40% of your samples lack DNA methylation data (common in heterogeneous cohorts), the model still trains without imputation.
- **Variance attribution**: The `calculate_variance_explained()` function tells you exactly how much variance each factor explains in each omic, enabling statements like "Factor 3 explains 12% of RNA variance but only 2% of methylation variance, suggesting a post-transcriptional regulatory mechanism."
- **Multi-group extension (MOFA+)**: The `+` in MOFA+ refers to the ability to model multiple groups of samples (e.g., batches, tissue types, time points) simultaneously, decomposing group-level vs. shared variation.
- **Interpretable factors**: Each factor has a loading vector per view. Inspecting the top positive/negative weights reveals the molecular features driving each axis of variation.

**Key limitations:**
- **No direct prediction**: MOFA+ does not produce a classifier. Factor scores can be used as features in downstream ML, but this is a two-step process.
- **Linear assumption**: Like PCA, MOFA+ assumes linear relationships between features and factors. Non-linear variation (e.g., bimodal expression distributions) may require non-linear methods.
- **Python dependency**: Training requires `mofapy2` via `reticulate`. The R package handles pre/post-processing but delegates ELBO optimization to Python.
- **Scale sensitivity**: Data from different omics often requires normalization to comparable scales before input (e.g., variance-stabilized RNA-seq, M-values for methylation).

**Typical workflow:**
```
Raw data → QC/normalize per omic → create_mofa() → set options →
run_mofa() → plot_variance_explained() → correlate factors with
clinical metadata → extract top weights → pathway enrichment
```

### DIABLO: Data Integration Analysis for Biomarker discovery with Latent cOmponents

**Mathematical foundation:**
DIABLO extends sparse Partial Least Squares Discriminant Analysis (sPLS-DA) to multiple data blocks. It finds components that simultaneously:
1. Maximize correlation between omics blocks (controlled by design matrix)
2. Maximize separation between outcome classes

The design matrix `A` (P × P, where P = number of omics) encodes the strength of inter-block correlation to impose. A value of 1 means full correlation maximization between two blocks; 0 means only discrimination matters.

```
Objective: max Σ_i,j A_ij · corr(X_i·a_i, X_j·a_j) + Σ_i disc(X_i·a_i, Y)
```

Sparsity is imposed by limiting the number of features retained per component per block (`keepX`), which is tuned by cross-validation.

**Key strengths:**
- **Integrated biomarker panel**: The sparse solution directly identifies a small, named set of features across all omics that jointly discriminate classes. This is immediately interpretable for clinical translation.
- **Cross-omic visualization**: The `circosPlot()` function shows correlations *between* selected features from different omics blocks—revealing, e.g., that a specific miRNA is inversely correlated with its target mRNA, both selected as discriminant features.
- **Built-in prediction**: `predict(model, newdata)` applies majority vote across components and blocks to classify new samples.
- **Performance estimation**: `perf()` runs repeated stratified cross-validation, giving AUC, BER, and class error rates with confidence intervals.

**Key limitations:**
- **Requires complete cases**: Every sample must have measurements for every omic block. Even a single missing block for a sample requires imputation or sample exclusion.
- **Supervised overfit risk**: If the sample-to-feature ratio is low, cross-validation is critical. Always use `perf()` to assess overfitting before interpreting biology.
- **Design matrix sensitivity**: Results can change meaningfully with different design matrix values. The recommendation of 0.1 (full design) vs 0 (null design) should be explored.
- **Linear only**: Like MOFA+, DIABLO assumes linear block relationships.

**Typical workflow:**
```
Raw data → QC/normalize per omic → define design matrix →
tune ncomp with perf() → tune keepX with tune.block.splsda() →
final model → plotLoadings() + circosPlot() + plotROC() →
validate on held-out test set
```

## Use Case Matrix

| Scenario | Recommended Method | Rationale |
|----------|-------------------|-----------|
| Discover CLL patient subtypes from multi-omics atlas | MOFA+ | Unsupervised, handles missing data in cohort studies |
| Build TCGA breast cancer subtype classifier | DIABLO | Supervised, explicit feature selection, test set validation |
| Explore batch effects across omics | MOFA+ (multi-group) | Decomposes between-batch vs. within-batch variation |
| Identify drug resistance biomarkers (responder vs non-responder) | DIABLO | Binary outcome, want minimal predictive panel |
| Integrate scRNA + scATAC single-cell | MOFA+ | Missing data by design; cells profiled by one method |
| Time-series multi-omics (pre/post treatment) | MOFA+ (multi-group) | Groups = time points; factors = dynamics |
| Regulatory network reconstruction | MOFA+ weights + network tools | Factor loadings as evidence of co-regulation |
| Clinical trial endpoint prediction | DIABLO | Labeled outcome, goal is deployable classifier |
| GWAS + transcriptomics QTL integration | MOFA+ or specialized (MEFISTO) | No label; want shared genetic-expression factors |
| Proteogenomics (tumor purity confounded) | MOFA+ | Purity captured as a factor and regressed out |

## Practical Notes

### Sample size guidelines
- MOFA+: n ≥ 20 per group; factors become unreliable below this
- DIABLO: n ≥ 30 per class; tune.block.splsda() needs ≥ 5-fold CV

### Feature selection before input
Both methods benefit from pre-filtering to the most variable features:
- **RNA-seq**: top 5,000 by mean absolute deviation (MAD) or Seurat's FindVariableFeatures
- **Methylation**: top CpGs by variance (exclude sex chromosomes and SNP-associated probes)
- **miRNA**: remove low-count miRNAs (< 10 reads in > 80% of samples)
- **Drug response**: AUC or IC50 after quality control on viability curves

### Data transformations
| Data type | MOFA+ recommended | DIABLO recommended |
|-----------|------------------|-------------------|
| RNA-seq counts | VST or log2(CPM+1) | log2(CPM+1) |
| Methylation arrays | M-values (logit of β) | M-values |
| miRNA counts | log2(CPM+1) | log2(CPM+1) |
| Drug AUC (0–1) | logit transformation | raw or logit |
| Mutations (binary) | Use Bernoulli likelihood option | Typically excluded |

### Combining both methods
In many real studies, MOFA+ is used first (exploratory) and DIABLO second (confirmatory). Example workflow:
1. Run MOFA+ on the full cohort → identify K=3 major factors
2. Factor 2 strongly correlates with treatment response (p < 0.001)
3. Extract top 50 features with high Factor 2 loadings per omic
4. Use these as the feature set for DIABLO, with response as outcome
5. DIABLO identifies the minimal discriminant panel from the MOFA-nominated features

This combination leverages the unsupervised discovery power of MOFA+ to reduce dimensionality, then the supervised specificity of DIABLO for clinical translation.
